# 博物馆的仓库

> **对应概念：** Artifact

## 故事

莉娜在榆树街小梅的面包坊隔壁开了一间恒温保管库。小梅丢掉那个周三蛋糕的惨剧，其实比配方丢失还严重。配方现在安全了——老王帮她整理好了笔记。但是蛋糕本身没了。横截面的照片、她画的风味分析图、打算给客户宴会留的样品——全部跟着当晚的垃圾一起扔掉了。

这时候莉娜出手了。「你的笔记本记的是你做了什么、结果怎么样，」她跟小梅说，「但配方不是蛋糕。真东西我来存。」从那以后，小梅笔记本里每一个 Run 都附带一个货架编号——指向莉娜保管库里实物的位置：蛋糕样品、照片、图表、原料包装标签。

莉娜的保管库里什么都有——不光是小梅的蛋糕，榆树街上所有需要保存实物（而不是只留描述）的人都往这儿存。每件东西上标着它属于哪个实验、哪个 Run。后来小梅团队里的新人阿毕开始自动记录每个蛋糕的详细信息，他每次也一定会附上一个指向莉娜保管库的地址。因为蛋糕的照片终究不是蛋糕本身。

## 概念解读

**Artifact** 就是 Run 产出的实际文件——模型权重、图表、特征统计、混淆矩阵、序列化的模型等。不是数字，不是标签，是真正的文件。

Artifact 和元数据（参数、指标、Tag）是分开存的。元数据存在 **backend store**（数据库）里，Artifact 存在 **artifact store**（对象存储）里——因为文件可能非常大。默认存在本地的 `mlruns` 目录，也可以配置成 S3、Azure Blob Storage 等。

数据库里只存一个 URI 地址，指向 Artifact 的实际存放位置。

::: tip 一句话总结
- **Artifact** = Run 产出的文件（模型、图表、数据等）
- 存在 artifact store 里，和元数据分开
- 数据库只存地址（URI），不存文件本身
:::

## 前端开发者参考

Run 详情页里的 **Artifact 浏览器**就是小丽的仓库。你要做的是展示一个文件树——用户可以展开文件夹、预览图片和文本、或者下载文件。你的数据来源是一个单独的 API（不是返回参数指标的那个），返回的是目录列表。

| 组件 | 对应什么 |
|------|---------|
| `ArtifactPage` / `ArtifactView` | 文件树浏览器 |
| 文件预览面板 | 在页面里直接展示图片、文本、HTML |
| 下载按钮 | 下载某个具体文件 |

### 数据长什么样

```typescript
// 文件树中的一个条目
{
  path: "model/weights.pkl",
  is_dir: false,
  file_size: 52428800  // 50 MB
}

// Run 上的 Artifact 地址
{
  run_id: "abc-def-456",
  artifact_uri: "s3://my-bucket/experiments/123/abc-def-456/artifacts"
}
```
